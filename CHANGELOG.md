## [1.0.0] - 2026-07-09

### Added
- Engineering Design Document ([`docs/EDD.md`](docs/EDD.md)) covering system architecture, module boundaries, database schema, API contracts, event catalog, C4 diagrams, sequence diagrams, payment state machine, repository structure, coding standards, implementation roadmap, and risk analysis.
- 12 Architecture Decision Records (`docs/adr/0001`–`0012`) covering the modular monolith, Kafka, PostgreSQL, Redis, transactional outbox, provider abstraction, idempotency strategy, double-entry ledger, multi-tenancy, authentication, webhook reconciliation, and observability decisions.
- Project README establishing PayFlow's identity as a Payment Orchestration Platform.

No application code has been written yet — this is the Phase 0 design milestone.

### Added (M0 — Bootstrap & Tooling)
- Maven reactor (`backend/pom.xml`) on Spring Boot 3.3.5 / Java 21 with two modules: `payflow-core` (the modular monolith skeleton) and `mock-provider-service` (standalone provider simulator skeleton).
- `payflow-core`: Spring Boot application class, `dev`/`test`/`prod` configuration profiles, Flyway baseline migration (`V1__init.sql`, enables `pgcrypto`), and a Testcontainers-backed context-load test that validates the app boots and migrations apply against a real PostgreSQL instance.
- `mock-provider-service`: Spring Boot application class and configuration, with a context-load test.
- `infra/docker-compose.yml`: local development stack (PostgreSQL 16, Redis 7, Kafka in KRaft mode) with health checks and named volumes.
- `.github/workflows/ci.yml`: GitHub Actions CI skeleton building and testing the backend reactor on JDK 21.
- Root `.gitignore` for Maven build output, IDE files, and future frontend artifacts.
- Verified manually end-to-end: `docker compose up` brings all three infra services to healthy; both Spring Boot jars start against that infra, Flyway applies migration V1 against real Postgres, and both `/actuator/health` endpoints report `UP`.

### Added (M1 — Tenancy & Security Core)
- `common` module: `PayFlowException` hierarchy (`ResourceNotFoundException`, `ConflictException`, `DomainValidationException`) with a Stripe-style JSON error envelope; `TenantContext`/`TenantContextHolder` (reads the current request's tenant off Spring Security's `Authentication.getDetails()`); AES-GCM `SymmetricEncryptor` for at-rest secrets.
- `infrastructure` module: `CorrelationIdFilter` (ahead of the security filter chain, so even 401/403 responses carry a trace id) and `GlobalExceptionHandler`.
- Flyway `V2__tenancy_and_security.sql`: `organizations`, `users`, `organization_members`, `api_keys`, `merchants`, `provider_accounts` — including a partial unique index enforcing at most one default provider account per merchant.
- `organization` module: signup (`POST /v1/organizations`, creates an org + OWNER in one transaction), API key issuance/list/revocation (`/v1/organizations/{id}/api-keys`) with Stripe-style `pf_live_`/`pf_test_` keys (BCrypt-hashed secret, indexed lookup prefix), and RBAC membership checks (`OrganizationAccessGuard`).
- `merchant` module: merchant CRUD and provider account creation with encrypted credentials, scoped under `/v1/organizations/{id}/merchants`.
- `security` module: stateless JWT access/refresh tokens (JJWT, HS512), `ApiKeyAuthenticationFilter` and `JwtAuthenticationFilter` (disambiguated by the `pf_` prefix), `SecurityConfig` (stateless session, JSON 401/403 bodies), `POST /v1/auth/login` and `/v1/auth/refresh`.
- Unit tests for `JwtService` and `ApiKeyService` (no Docker required) plus a Testcontainers-backed `AuthFlowIntegrationTest` covering signup, login, refresh, cross-tenant/cross-principal rejection, and the provider-account default-flip.
- Fixed a real Hibernate flush-ordering bug found during manual verification: inserting a new default provider account raced the update that cleared the previous default against the `is_default` partial unique index (inserts flush before updates by default) — fixed with an explicit `flush()` between the two writes.
- Fixed a Flyway/Hibernate schema mismatch (`default_currency` declared `CHAR(3)`, but a plain `String` field validates against `VARCHAR`) caught the same way — via a real Postgres, not a mock.
- Corrected the module dependency table in the EDD (§3): `security` depends on `organization` (to validate API keys and authenticate users), not the other way around; `organization` only needs the `spring-security-crypto` library for password hashing, not the `security` module itself.

### Added (M2 — Payment Lifecycle)
- Relocated `ProviderCode` from `merchant.domain` to `common.provider`, since both `merchant` and `payment` need it and neither should reach into the other's domain package.
- `provider` module: `ProviderClient`/`ProviderRegistry` abstraction (ADR-0006), `MockProviderClient` calling the separate Mock Provider service over real HTTP via Spring's `RestClient`.
- Flyway `V3__payments.sql`: `payments` (`provider_account_id`/`provider_code`/`provider_reference` nullable - resolved at authorize time, not create time, per the EDD's own API contract) and `payment_state_transitions`, with check constraints on all amount columns.
- `payment` module: `Payment`/`PaymentStateTransition` entities (metadata stored as native `jsonb` via Hibernate 6's `@JdbcTypeCode(SqlTypes.JSON)`), the `PaymentStateMachine` (pure, Spring-free, directly unit-tested), and `PaymentService` covering create/authorize/capture/get/list. Merchant-facing API at `/v1/payments/**`, API-key authenticated only.
- `mock-provider-service`: in-memory `ChargeStore` and `POST /provider/v1/charges` / `POST /provider/v1/charges/{id}/capture` — deterministic (always succeeds) for M2; configurable latency/failure simulation is M11.
- 27 unit tests (state machine + service-layer with Mockito) plus a Testcontainers integration test covering the create/get/list/validation/tenant-isolation surface over real HTTP and Postgres (authorize/capture need the separate live Mock Provider service, so those are covered by manual verification instead, documented below).
- **Fixed two real bugs found by writing tests, not by inspection**, before either shipped:
  - `authorize()` and `capture()` were calling the provider *before* validating the state transition was legal — meaning calling `authorize()` twice on an already-authorized payment would have created a **second charge at the provider** before finally failing. Fixed by validating the transition first, so an illegal call never reaches the provider at all (verified with a Mockito `never()` assertion).
  - Illegal-transition errors (double-capture, double-authorize) were returning 400 instead of the intended 409: Spring resolves competing `@RestControllerAdvice` beans by checking advice **beans** in order and using the first one with any applicable handler, not the most specific handler across all beans — so the generic `GlobalExceptionHandler`'s `PayFlowException` catch-all was winning over the more specific `PaymentExceptionHandler`. Fixed by making the generic handler explicitly `@Order(Ordered.LOWEST_PRECEDENCE)` and the specific one `HIGHEST_PRECEDENCE`.
- Fixed an EDD inconsistency caught while writing the create-payment DTO: the ER diagram for `payments` had no `description` column, but the API contract's own example request body included one. Added the column since the migration hadn't been applied anywhere yet.
- Manually verified end-to-end against both live services (16 scenarios): signup → merchant → provider account → API key → create → authorize → capture → double-authorize (409) → double-capture (409) → over-capture (400) → get detail with full transition history → list filtered by merchant/status → cross-org merchant rejection (404) → JWT session rejected from merchant-facing endpoints (403).

### Added (M3 — Idempotency)
- Added Redis to the stack (`spring-boot-starter-data-redis`) - the first milestone to actually need it.
- Flyway `V4__idempotency_keys.sql`: `idempotency_keys`, unique on `(organization_id, key)`, `response_body` stored as plain `TEXT` rather than `jsonb` (it's an opaque blob the app never queries structurally, so `jsonb` would only have added Hibernate JSON-mapping risk for no benefit).
- `idempotency` module: `IdempotencyKey` entity, sealed `IdempotencyCheckResult` (`Proceed`/`Replay`/`InProgress`/`FingerprintMismatch`), `IdempotencyGuard`/`IdempotencyService` — Postgres is the durable source of truth, Redis a fast-path cache that degrades to the database on any failure (ADR-0004). Concurrent duplicate inserts are resolved by catching the unique-constraint violation rather than an explicit `SELECT ... FOR UPDATE` — simpler, same guarantee, no row-lock hold time across a request.
- `IdempotencyFilter`: a single cross-cutting filter (not per-controller code) that activates on any mutating request carrying an `Idempotency-Key` header, wired in after both auth filters via `SecurityConfig`. Reads the request body once via a custom `CachedBodyHttpServletRequest` so both the fingerprint computation and the controller's own parsing can consume it. Captures the response via `ContentCachingResponseWrapper` for replay; a response `< 500` is cached, a `>= 500` abandons the record instead so a retry starts fresh rather than being stuck replaying a transient failure forever.
- **Scope decision:** idempotency applies to API-key authenticated requests only. JWT-authenticated dashboard sessions have no `organizationId` on their `TenantContext` (a user can belong to multiple orgs), and a human clicking a button twice is a different problem than a payment integration retrying a network timeout — which is what ADR-0007 is actually about. Verified manually: the same `Idempotency-Key` sent twice on a JWT-authenticated merchant-creation call correctly creates two merchants, not one.
- **Scope decision:** `payments.idempotency_key_id` (present in the original ER diagram) was deliberately not added — same reasoning as M2's `description` fix in reverse: there's no current consumer that needs the linkage (no audit view exists yet showing "which request created this payment"), so adding an unpopulated column now would be exactly the kind of speculative schema growth the coding standards warn against. Straightforward additive migration whenever a real consumer shows up.
- Refactored the Testcontainers test setup: extracted a shared `AbstractIntegrationTest` (Postgres + Redis containers, plus the tenant-provisioning helper that was duplicated in `PaymentLifecycleIntegrationTest`) since every full-context test now needs both containers - four near-identical copies would have been worse than one shared base.
- 9 `IdempotencyService` unit tests (including a Mockito-simulated Redis outage proving the ADR-0004 degrade-to-database guarantee actually holds) plus a Testcontainers `IdempotencyIntegrationTest` covering dedup, fingerprint-mismatch rejection, and no-key independence over real HTTP, Postgres, and Redis.
- Manually verified end-to-end with a live Redis this time (13 scenarios) — including confirming actual cache entries in Redis via `redis-cli KEYS`, not just correct HTTP responses. No new bugs found on this pass.

### Added (M4 — Ledger)
- Flyway `V5__ledger.sql`: `ledger_accounts` (unique per `(organization_id, code)`), `ledger_transactions`, `ledger_entries` — plus two real pieces of database engineering, not just tables:
  - A **deferred constraint trigger** enforcing ADR-0008's "entries under one `ledger_transaction_id` net to zero per currency" invariant. A plain `CHECK` constraint can't aggregate across rows, so this uses `CREATE CONSTRAINT TRIGGER ... DEFERRABLE INITIALLY DEFERRED`, which validates once at commit time — after both the debit and credit legs of a posting have been inserted, not after each individual row.
  - `BEFORE UPDATE`/`BEFORE DELETE` triggers making `ledger_entries` genuinely append-only. A `REVOKE UPDATE, DELETE FROM PUBLIC` (the literal wording in ADR-0008) would **not** actually have restricted anything: Postgres table owners retain full privileges regardless of grants, and the application connects as the same role that ran the migrations. The triggers reject the operation unconditionally, for any role, which is what "no one can edit history" actually requires.
- `ledger` module: `LedgerAccount`/`LedgerTransaction`/`LedgerEntry` entities, `LedgerAccountCode` (couples each well-known account to its correct `LedgerAccountType` so `merchant_payable` can never accidentally be created as an `ASSET`), `LedgerService` (posting) + `LedgerQueryService` (reads) — both implemented by one `LedgerServiceImpl`, matching the two interfaces the EDD's own module table already committed to. `PaymentService.capture()` calls `LedgerService.postCapture()` inside its existing `@Transactional` method, so "captured but not yet in the ledger" stays an unreachable state rather than a race to defend against.
- `GET /v1/ledger/entries` (API-key authenticated, filters by `paymentId`/`createdAfter`/`createdBefore`).
- **Scope decision:** `ledger_transactions.refund_id` from the original ER diagram was not added — same reasoning as M2/M3's deferred-column decisions: the `refunds` table doesn't exist until M5, so there's nothing to reference yet. Additive `ALTER TABLE` when M5 needs it.
- **Fixed two real bugs, both only surfacing once the code actually ran against real Postgres** (Mockito mocks can't catch either):
  - `postCapture()` needs to look up-or-create two accounts, then insert a transaction and two entries, all in one method. The account-lazy-creation race handling originally used `@Transactional(propagation = NESTED)` on a separate bean, reasoning (correctly, per the code review) that self-invocation would silently skip the propagation if it were a private method. What the reasoning missed: Spring's `JpaTransactionManager` doesn't support real savepoints at all (`JpaDialect does not support savepoints`), so `NESTED` throws immediately, regardless of the self-invocation fix. Switched to `Propagation.REQUIRES_NEW` — a genuinely separate transaction, which needs no savepoint support and is a better semantic fit anyway: ledger accounts are idempotent chart-of-accounts metadata, not part of the specific capture's financial data, so there's no correctness reason to tie their creation to the same transaction.
  - `GET /v1/ledger/entries` 500'd with `could not determine data type of parameter`. The `(:paymentId IS NULL OR e.ledgerTransaction.paymentId = :paymentId)` pattern — already used successfully in `PaymentRepository.search()` for direct root-entity fields — fails specifically over a joined association path; Postgres can't infer the bare `? IS NULL` parameter's type with no other context, and apparently Hibernate's SQL generation doesn't supply one for this shape of query. Fixed with a native query and explicit `CAST(:param AS uuid/timestamptz)` on every optionally-null parameter, which pins the type unambiguously regardless of Hibernate's translation.
- Both bugs were caught by the same manual curl walkthrough that's caught every real bug so far this project (M1's schema mismatch, M2's flush-ordering race and advice-precedence bug) — this pass also directly exercised the two triggers via `psql` (confirmed `UPDATE`/`DELETE` on `ledger_entries` rejected with the exact expected error, and a deliberately unbalanced insert rejected at `COMMIT` with the exact expected imbalance amount), which no HTTP-level test could have proven.

### Added (M5 — Refunds)
- Flyway `V6__refunds.sql`: `refunds` table, plus the `ledger_transactions.refund_id` column deferred from M4 (`ALTER TABLE`, nullable, `NULL` for `CAPTURE`-type transactions) — `refunds` has to exist first, so this had to wait until now regardless.
- **First genuine cross-module consumer, exactly as M2 anticipated:** `refund` depends on `payment` (per the EDD's own module table), but must never reach into `payment.domain`/`payment.persistence`. Added `PaymentRefundSupport` (`loadForRefund` / `applyRefund`) — a narrowly-scoped interface implemented by the existing `PaymentService`, alongside its current public methods, the same way `LedgerServiceImpl` already implements two interfaces for two different callers. `loadForRefund` fails fast with a clear message if the payment isn't `CAPTURED`/`PARTIALLY_REFUNDED`, before an amount is even computed; `applyRefund` keeps all `PaymentStateMachine` validation and transition recording inside the payment module, where it belongs — `refund` never imports `PaymentStateMachine` or even knows it exists.
- `provider` module: `ProviderClient.refund()` (additive, per ADR-0006's own extensibility promise), `MockProviderClient` implementation calling the Mock Provider's new `POST /provider/v1/charges/{id}/refund` — deterministic success for a `CAPTURED` charge, matching M2's capture endpoint's own "always succeeds, chaos simulation is M11" scope.
- `refund` module: `Refund` entity (resolved synchronously in one call, unlike `Payment`'s multi-step lifecycle — the provider has already confirmed the refund by construction time, so there's no separate "mark succeeded" step and no state-transition audit table), `RefundService.createRefund()` (validates the amount against `capturedAmount - refundedAmount`, calls the provider, then — in the same transaction — saves the refund row, calls `PaymentRefundSupport.applyRefund()`, and calls `LedgerService.postRefund()`, per ADR-0008's "must never be a reachable state without the ledger posting"). `POST /v1/payments/{id}/refunds` (full or partial; omitting `amount` refunds the remaining balance) and `GET /v1/refunds/{id}`, both API-key authenticated.
- `LedgerService.postRefund()`: posts the reverse of a capture (`Dr Merchant Payable` / `Cr Provider Settlement Receivable`, per ADR-0008), linked to the refund via the now-populated `refund_id`.
- A failed provider refund persists nothing at all — no `Refund` row, no payment mutation, no ledger posting — mirroring exactly how `PaymentService.capture()` already treats a failed capture: safe to retry, never a partially-applied state.
- **Scope decision:** `Idempotency-Key` support for refund creation needed zero new code — `IdempotencyFilter` is path-agnostic (any mutating, API-key-authenticated request), so it started covering `POST /v1/payments/{id}/refunds` automatically. Verified manually below.
- **Scope decision:** the `REFUNDS` ER diagram in the EDD had no `created_at` column, unlike every other audited table (`payments`, `ledger_transactions`, `payment_state_transitions`) — an oversight, not a deliberate omission, so added it (same category of fix as M2's `payments.description`).
- 7 `RefundServiceTest` unit tests (full/partial/multi-partial refunds, over-refund rejection, zero-amount rejection, non-refundable payment state rejection, provider-failure rejection with nothing persisted) plus 4 new `PaymentServiceTest` cases for `loadForRefund`/`applyRefund`, a `LedgerServiceImplTest` case for `postRefund`'s reversed entries, and a `LedgerIntegrationTest` case verifying the real `refund_id` foreign key and reversed account codes against actual Postgres.
- Manually verified end-to-end against both live services: create → authorize → capture → partial refund ($30) → over-refund attempt (400) → remaining-balance refund via omitted `amount` ($70, payment reaches `REFUNDED`) → refund attempt on an already-`REFUNDED` payment (400) → refund attempt on an `AUTHORIZED`-only payment (400, rejected before the provider is ever called) → idempotent refund retry (same `Idempotency-Key` + body replays the identical refund, confirmed via `refundedAmount` staying at the requested amount, not doubling). Also confirmed via `psql` that `ledger_transactions.refund_id` is `NULL` for the capture row and correctly set for both refund rows, and that the six resulting ledger entries net to zero per transaction. No new bugs found on this pass.

### Added (M6 — Outbox + Kafka)
- Flyway `V7__outbox.sql`: `outbox_events` (`payload` plain `TEXT`, not `jsonb` — the poller only ever republishes it verbatim to Kafka, never queries into it structurally, same reasoning as `idempotency_keys.response_body` in V4) and `dead_letter_events`.
- **Scope decision:** the EDD's `OUTBOX_EVENTS`/`DEAD_LETTER_EVENTS` ER diagrams had no `created_at` at all, and `outbox_events` had no `updated_at` either — unlike every other audited table. Added both: `created_at` is load-bearing (the poller's `ORDER BY created_at` needs it), and `updated_at` is genuinely meaningful here in a way it isn't for `ledger_entries` — `outbox_events` is mutable (status flips `PENDING`→`PUBLISHED`, `retry_count` increments), not append-only.
- `outbox` module: `OutboxEvent`/`DeadLetterEvent` entities, `OutboxWriter` (the only way any other module writes an `outbox_events` row — always called from inside the caller's own `@Transactional` method, so the row commits atomically with the business row it describes, per ADR-0005), `OutboxPublisher` — a `@Scheduled` poller using `SELECT ... FOR UPDATE SKIP LOCKED` (lets multiple instances poll concurrently without contending on the same rows) to relay `PENDING` rows to Kafka, marking `PUBLISHED` on success or incrementing `retry_count` on failure, moving to `dead_letter_events` once `max_retries` is exceeded.
- `common.event`: `PaymentEventPayload`/`RefundEventPayload`/`LedgerEventPayload`/`OutboxTopics` — `common` explicitly owns "event contracts" per the EDD's own module table, and must have zero dependency on any other module, so these payloads use plain `String status`/`transactionType` fields rather than importing `payment.domain.PaymentStatus` or `ledger.domain.LedgerTransactionType`.
- Wired into `PaymentService` (`payment.created`, `payment.authorized`, `payment.authorization_failed`, `payment.captured`) and `RefundService` (`refund.succeeded`), plus `ledger.transaction_recorded` emitted by *both* of them right after calling `LedgerService` — not by the ledger module itself, which the EDD's module table restricts to depending only on `common`, never `outbox`.
- **Scope decision, several event types from the EDD's own catalog deliberately not emitted yet:** `payment.capture_failed` and `refund.failed` have no producing code path — both a failed capture and a failed refund throw and roll back their entire transaction before anything commits (established in M2/M5), so there is no committed row to attach an event to. `refund.created` doesn't either — a refund resolves synchronously in one call, so there's no distinct "created" moment before "succeeded". `payment.expired`/`payment.canceled` have no producer until M9's scheduled sweep exists. All four are additive whenever their producing code exists.
- 4 `OutboxWriterImplTest`/`OutboxPublisherTest` unit tests, 6 new `PaymentServiceTest`/`RefundServiceTest` assertions verifying the right events fire (and don't fire on failure), and a Testcontainers `OutboxIntegrationTest` — `AbstractIntegrationTest` gains a `KafkaContainer` alongside Postgres and Redis — verifying a real publish end-to-end via a raw `KafkaConsumer` against the real broker. The test profile sets `poll-interval-ms` to an hour so the background schedule never fires during a test run; tests call `relayPendingEvents()` directly instead, which avoids any race between the schedule and the test's own assertions.
- **Fixed two real bugs, both found only by killing the Kafka container mid-flow, not by any test:**
  - `dead_letter_events.error_reason` came back empty for the single most common failure case: `java.util.concurrent.TimeoutException` from `Future.get(timeout)` has a `null` `getMessage()` by construction, which defeated the entire point of the column (ADR-0005: "for operator visibility"). Fixed by falling back to the exception's simple class name whenever the message is `null`.
  - Each failed publish attempt was taking **~60 seconds**, not the intended ~5: `KafkaTemplate.send()` itself blocks synchronously while the producer fetches initial cluster metadata, before it ever returns the future `OutboxPublisher`'s own `.get(5, SECONDS)` bounds — that initial block is governed by the Kafka client's own `max.block.ms`, which defaults to 60000. Since `relayPendingEvents()` holds `FOR UPDATE SKIP LOCKED` row locks for its entire transaction, this meant one unreachable broker could hold outbox rows locked for a full minute per retry. Fixed by setting `spring.kafka.producer.properties.max.block.ms: 3000`, below `OutboxPublisher`'s own timeout, so that timeout is actually the effective per-attempt bound. Confirmed the fix directly: exhausting 5 retries and reaching `dead_letter_events` went from an estimated 5 minutes to ~31 seconds.
- Manually verified end-to-end against Postgres, Redis, and a real Kafka broker: full create → authorize → capture → partial refund flow produced exactly the 6 expected `outbox_events` rows, all flipped `PENDING`→`PUBLISHED` by the scheduled poller within ~2 seconds; consumed all three topics directly via `kafka-console-consumer.sh` and confirmed the partition key (`payment_id`/`organization_id`) and payload shape on each; killed the Kafka container mid-flow and confirmed retry_count incrementing, then dead-lettering with a real error message, then a fresh payment publishing cleanly with zero retries once Kafka came back.

### Added (M7 — Inbound Webhooks + Reconciliation)
- Flyway `V8__inbound_webhook_events.sql`: `inbound_webhook_events` (`payload` plain `TEXT`, same reasoning as `outbox_events`/`idempotency_keys`) plus `UNIQUE (provider_code, provider_reference)` on `payments` — reconciliation looks a payment up by that pair, so it has to actually be unique, not just improbable-to-collide (Postgres allows unlimited `NULL`s under a `UNIQUE` constraint, so pre-authorization payments with no reference yet are unaffected).
- **Fixed a real EDD inconsistency before implementing:** the C4 diagram already showed `webhook → payment` ("Reconciles state from provider webhook"), and ADR-0011 requires it structurally, but section 3's module dependency table omitted `payment` from `webhook`'s dependency list. Added it.
- `security.hmac`: `HmacSigner`/`HmacSha256Signer` — a generic HMAC-SHA256 sign/verify primitive (security owns "HMAC sign/verify" per the EDD), constant-time comparison via `MessageDigest.isEqual`. Deliberately knows nothing about webhooks specifically, so the same primitive will serve M8's outbound merchant webhook signing later.
- `payment` module gains `PaymentReconciliationSupport.reconcileCaptureConfirmation()` — the only surface `webhook` is allowed to call into `payment` through. `PaymentService.capture()` was refactored to extract the shared `applyCapture()` tail (state transition, ledger posting, outbox emission) so the synchronous and webhook-driven paths post *identical* ledger entries and events, per ADR-0011 ("must both funnel through the same state transition and ledger-posting logic").
- **Scope decision:** only capture confirmation is reconciled. Authorize and refund reconciliation both hit a gap capture doesn't have — `PaymentService.authorize()` and `RefundService.createRefund()` both roll back their entire transaction on a provider communication failure (established in M2/M5), persisting nothing, so a lost sync response for either leaves no committed row for a webhook to *confirm* — reconciling them would mean creating state from the webhook alone, which is a materially different operation the EDD's own worked example (section 7.2, capture-specific) doesn't cover. `charge.authorized`/`charge.refunded` webhooks are still received, signature-verified, deduplicated, and recorded — just not acted on yet.
- **Scope decision:** a single shared secret (`payflow.provider.mock.webhook-secret`, matched by mock-provider-service's own `payflow.webhook.secret`) authenticates all Mock Provider webhooks, not a secret stored per `provider_account` as the EDD's prose literally suggests. Mock Provider's `ChargeStore` has no concept of merchant/account association for a charge at all today; wiring a real per-account secret through would mean threading it through every `ProviderClient` request record for a provider that doesn't yet model multi-tenant credentials. One honest secret for "the one provider that exists" now, a natural extension point (mirroring `ProviderClient`/`ProviderRegistry`'s own per-provider isolation) whenever a second real provider needs its own.
- `webhook.inbound` module: `InboundWebhookEvent` entity, `InboundWebhookProcessor`/`InboundWebhookService` (verify signature → dedupe → reconcile), `POST /v1/webhooks/providers/{providerCode}` (`permitAll` in `SecurityConfig` — providers have neither an API key nor a JWT). Signature format mirrors the EDD's own outbound convention (`t=<ts>,v1=<hex hmac-sha256>`), including a 5-minute timestamp tolerance window against replay.
- `mock-provider-service` gains `WebhookSender`: fires an async, signed webhook after every charge operation (create/capture/refund), regardless of the synchronous response, matching EDD section 5.3. Fire-and-forget on the common `ForkJoinPool` — a low-volume simulation service, not worth a dedicated executor. A small, deliberate duplication of `HmacSha256Signer`'s ~10 lines rather than a shared library between two intentionally independent deployables.
- 5 `HmacSha256SignerTest` + 7 `InboundWebhookServiceTest` unit tests (valid/invalid/malformed/stale signatures, dedup, capture reconciliation dispatch, unsupported provider) plus 4 new `PaymentServiceTest` cases for `reconcileCaptureConfirmation` (applies, no-ops when already captured, no-ops when not found, no-ops when the webhook amount exceeds what's still capturable).
- **Fixed three real bugs, none caught by any test — only by running both live services together and deliberately breaking things:**
  - `POST /v1/webhooks/providers/mock` 500'd on every real delivery. The EDD's own documented endpoint is lowercase (`.../providers/mock`), a normal REST convention, while `ProviderCode`'s enum literals are uppercase, a normal Java convention — Spring's default enum path-variable converter is case-sensitive and satisfies neither convention by bending to the other. Fixed by binding the path variable as a plain `String` and converting explicitly with `.toUpperCase()`.
  - **A genuine double-ledger-posting bug**, found by re-running the exact happy path after fixing the bug above: a synchronous `capture()` and the webhook it triggers can race — the webhook can reach `InboundWebhookController` and start its own `reconcileCaptureConfirmation()` transaction *before* the synchronous capture's own transaction commits. Under Postgres's default READ COMMITTED isolation, a plain `SELECT` in either transaction can see the payment as still `AUTHORIZED` while the other is mid-flight, so both legitimately (from their own point of view) apply the capture — two `ledger_transactions`, two `payment.captured` outbox events, for one real capture. Confirmed via direct SQL inspection: 4 `ledger_entries` and 2 `payment.captured` outbox rows for a single capture. Fixed with `SELECT ... FOR UPDATE` (`PaymentRepository.lockByIdAndOrganizationId`/`lockByProviderCodeAndProviderReference`) on every payment load that precedes a mutation — `authorize()`, `capture()`, `loadForRefund()`/`applyRefund()`, and `reconcileCaptureConfirmation()` — so the second of any two concurrent mutations blocks until the first commits, then correctly observes the new state and no-ops. Reverified the identical scenario: exactly 3 transitions, 2 ledger entries, 1 outbox event.
  - Fixing the race above with a `lockBy...`-named repository method crashed the app at startup: Spring Data JPA only recognizes `find`/`read`/`get`/`query`/`count`/`exists`/`delete`/`remove` as query-derivation prefixes, so `lockByIdAndOrganizationId` was parsed as a property path (`lockByIdAndOrganizationId` → property `lockByIdAndOrganizationId` on `Payment`, which doesn't exist) rather than treating `@Lock` as the modifier it is. Fixed with an explicit `@Query`, which makes the method name irrelevant to parsing.
  - A fourth issue surfaced testing webhook retry delivery directly: **every legitimate webhook retry 500'd** instead of being a no-op. Catching `DataIntegrityViolationException` after a failed `saveAndFlush()` doesn't actually recover the transaction — Postgres aborts the entire transaction on any statement failure, so the Java-level catch succeeds but the subsequent commit throws `UnexpectedRollbackException` regardless. Fixed with an explicit `existsByProviderCodeAndProviderEventId` check before the insert, avoiding the constraint violation (and the transaction-abort it causes) for the expected case entirely; a true simultaneous double-delivery race still 500s, which is an acceptable, retry-friendly outcome given webhook delivery is expected to retry on error.
- Manually verified end-to-end against both live services and real Postgres/Redis/Kafka: full create → authorize → capture (happy path, webhook arrives after the sync path already committed, correctly no-ops) → a capture applied *only* by calling the Mock Provider directly (bypassing PayFlow's own `capture()` entirely) reconciled purely from the async webhook, `actor=PROVIDER_WEBHOOK`, correct ledger posting → forged signature rejected with 401 → replayed webhook delivery (3x, same event) correctly idempotent → full create→authorize→capture→partial-refund flow producing exactly the right transition count, ledger entry count, and recorded webhook events with no duplication anywhere.

### Added (M9 — Scheduled Jobs)
- **Fixed a real gap in the EDD before this milestone could start:** the provider-facing API table (§5.3) never actually specified a way to poll a charge's current status — every prior milestone had gotten by on the provider's own inbound webhook alone. `ReconciliationSweeper` (below) needs exactly that as its backstop, so added `GET /provider/v1/charges/{id}` to the EDD first, then implemented it: `ProviderClient.checkStatus()` (additive, same extensibility pattern as M5's `refund()`), `MockProviderClient` calling the new endpoint, `mock-provider-service`'s `ChargeController.getStatus()`. `ProviderChargeStatusResult` carries `amount`/`currency` alongside status (not just a bare enum) so the sweeper can reconcile through the same amount-bearing path M7's inbound webhook uses, rather than guessing or blindly trusting `payment.getAmount()` — wrong for a partial capture.
- New `<module>.scheduled` sub-package convention, used by all five jobs below: the job class owns "find candidates, loop, isolate one failure from the rest of the batch"; the actual per-aggregate mutation stays in the owning module's existing `application`-layer service, unchanged.
- `ExpiredAuthorizationJob` (`payment.scheduled`): the "`AUTHORIZED → EXPIRED`: auth window elapsed" edge from the EDD's own state diagram (§8), finally wired up. Purely time-based, never calls the provider — a real auth hold expiring is provider-side behavior PayFlow mirrors locally, not something to cross-check. `PaymentService.expireAuthorization()` uses a new unscoped `PaymentRepository.lockById()` (no `organizationId` — a system-wide sweep has no single tenant's request context) and finally exercises `PaymentActor.SCHEDULED_JOB`, present in the domain model since M2 but unused until now. Also finally wires the `payment.expired` outbox event, explicitly deferred since M6.
- `ReconciliationSweeper` (`payment.scheduled`): the backstop ADR-0011 requires — cross-checks payments stuck `AUTHORIZED` past a short window directly against the provider via the new `checkStatus()`, and reuses `PaymentReconciliationSupport.reconcileCaptureConfirmation()`, the *exact* method M7's inbound webhook path calls, so a payment recovered here posts an identical ledger entry and outbox event (ADR-0011: sync, webhook, and sweep paths "must both funnel through the same state transition and ledger-posting logic"). Same query shape as `ExpiredAuthorizationJob` (`findByStatusAndAuthorizedAtBefore`), deliberately different windows: 15 minutes by default here (catch a lost webhook fast) vs. 168 hours there (mirror a real provider's auth-hold lifetime) — a `NOT_FOUND` provider response is logged, never acted on, since a transient provider-side hiccup shouldn't corrupt local state.
- `DeadLetterWriter`/`DeadLetterWriterImpl` (`outbox.application`): extracted from `OutboxPublisher`'s direct `DeadLetterEventRepository` use into a proper public interface, so `WebhookRetryJob` (below) can dead-letter through it too without a module-boundary violation — `webhook` already declared a dependency on `outbox` in the EDD's module table for exactly this, unused until now.
- `OutboxCleanupJob` (`outbox.scheduled`) and `IdempotencyKeyCleanupJob` (`idempotency.scheduled`): cron-based, delete `PUBLISHED` outbox rows past a retention window and `idempotency_keys` rows past their own `expires_at`, respectively. `OutboxCleanupJob` only ever touches `PUBLISHED` rows — a `PENDING` row old enough to match the cutoff is a stuck event an operator needs to see, not something to clean up silently.
- `WebhookDeliveryAttempter` (`webhook.outbound.application`): the sign+POST+interpret-response logic extracted from `WebhookDispatcher.deliver()` into its own component returning a `DeliveryOutcome`, so a retried delivery is indistinguishable, on the wire, from the original M8 attempt. Never throws — a communication failure becomes a failed `DeliveryOutcome`, not an exception either caller has to catch.
- `WebhookRetryJob` (`webhook.outbound.scheduled`): EDD §7.5's retry/DLQ sequence, implemented exactly — selects `FAILED` rows past their own `next_retry_at` (`FOR UPDATE SKIP LOCKED`, same reasoning as M6's `OutboxEventRepository.lockNextBatch`, so multiple app instances never redeliver the same webhook twice in one cycle), retries via the shared attempter, and on failure either reschedules with `WebhookDelivery.incrementAttemptAndScheduleRetry()` or, once `attempt_number` would reach `max_attempts`, calls the new `markExhausted()` and dead-letters through `DeadLetterWriter`. Handles one edge case the EDD sequence doesn't spell out: if the owning endpoint was disabled since the last failure, `pauseRetry()` nulls `next_retry_at` (so the row stops matching the query) without dead-lettering it — a disabled endpoint isn't a delivery failure.
- 6 new test classes (`ExpiredAuthorizationJobTest`, `ReconciliationSweeperTest`, `OutboxCleanupJobTest`, `IdempotencyKeyCleanupJobTest`, `WebhookRetryJobTest`, `WebhookDeliveryAttempterTest`) plus 3 new `PaymentServiceTest` cases for `expireAuthorization`. Refactoring `WebhookDispatcher` to use `WebhookDeliveryAttempter` required updating `WebhookDispatcherTest` and `OutboxPublisherTest` to mock the new seams instead of the old direct repository/client fields — both still pass, no behavior change.
- Manually verified all five jobs against both live services and real Postgres/Redis/Kafka, with each job's interval shortened via environment-variable overrides (never by changing a job's actual selection logic) so a live run didn't require waiting out a real 168-hour or 24-hour window: `ExpiredAuthorizationJob` expired a fresh authorization on its very first sweep; `ReconciliationSweeper` recovered a payment captured directly at the mock provider while `payflow-core` was deliberately stopped (a genuinely lost webhook, not a simulated one) back to `CAPTURED` with `actor=PROVIDER_WEBHOOK`; `OutboxCleanupJob` and `IdempotencyKeyCleanupJob` each deleted a real row on their next cron tick; `WebhookRetryJob` retried a delivery against a real closed TCP port through to `EXHAUSTED` plus a `dead_letter_events` row, and separately confirmed the disabled-endpoint pause path leaves `next_retry_at` null with no dead-letter written.
- **Found one real interaction worth documenting, not a bug:** with `ExpiredAuthorizationJob`'s and `ReconciliationSweeper`'s windows both shortened to effectively zero for a first verification pass, `ExpiredAuthorizationJob` won the race and expired a payment before the sweeper's next tick could reconcile it — because both jobs share the same candidate query, an equal-or-shorter expiry window than the reconciliation window lets the "give up" job beat the "catch a lost webhook" job to the same row. Not reachable with the shipped defaults (168 hours vs. 15 minutes gives the sweeper roughly 672 chances first), but confirms the two windows are not independent tuning knobs — `reconciliation.stale-window-minutes` must stay meaningfully shorter than `expired-authorization.auth-window-hours` for the backstop to actually backstop anything.

### Added (M11 — Mock Provider Hardening)
- Genuinely two-sided, both named in the roadmap and EDD: Mock Provider gets configurable latency/failure simulation (EDD §5.3: "every call has randomized latency 50-2000ms, a configurable failure rate"), and `ProviderClient` calls get the retry-with-backoff and circuit breaker the EDD's own risk table (§12) already called for but nothing had implemented — "Exponential backoff with jitter on ... provider client retries; circuit breaker around `ProviderClient`." Chaos without resilience to test against it would have been pointless; implementing both together is what actually exercises either one.
- **Fixed a real, dangerous gap before enabling retry on `authorize()` at all:** `ChargeStore.create()` accepted a `merchantReference` in its request shape but silently dropped it — every call, including a retried one, created a brand-new charge. A retry after a lost response (exactly what M11's own `Retry` now does) would have created a second, orphaned charge at the provider with no local record of it at all — worse than the capture/refund case, where a retried already-applied operation just hits a 409 the `ReconciliationSweeper` backstop (M9) eventually corrects. `ChargeStore` now dedupes by `merchantReference` via `computeIfAbsent` (atomic under concurrent retries), matching how a real provider's idempotency-key handling works.
- `ChaosFilter` (`mock-provider-service`, `OncePerRequestFilter`, matching the project's established filter-based cross-cutting pattern): applies randomized latency and a configurable failure rate ahead of every `/provider/v1/**` request, scoped away from actuator endpoints deliberately — chaos on a liveness/readiness probe would be actively harmful, not realistic, and could get a healthy instance killed by an orchestrator. `ChaosProperties`: latency defaults *on* at the EDD's real 50-2000ms range (slower-but-correct doesn't threaten test determinism), but `failureRate` defaults to *0* — a nonzero default would have made every prior milestone's curl-based manual-verification workflow flaky, which is exactly what that workflow has depended on staying deterministic to keep catching real bugs. Both fully overridable via env vars for a dedicated chaos exercise.
- `resilience4j-spring-boot3` added to `payflow-core`. A single named `providerClient` Resilience4j instance (retry: 3 attempts, exponential backoff with jitter, retrying only `ProviderCommunicationException`; circuit breaker: count-based sliding window, opens at a 50% failure rate over the last 10 calls) — one shared policy for now, since only the Mock adapter exists; per-provider-code naming is a natural extension once a second real provider does.
- `ResilientProviderClient` (`provider` module): wraps any `ProviderClient` with the `providerClient` retry + circuit breaker, so every adapter — Mock today, a real provider later — gets the same resilience without reimplementing it, matching ADR-0006's "payment/refund depend only on `ProviderClient`" intent. `CircuitBreaker` wraps `Retry`, not the reverse — Resilience4j's own documented composition order, so an open circuit fails fast without even attempting a retry cycle. `CallNotPermittedException` (thrown when the circuit is open) is caught and rewrapped as `ProviderCommunicationException`, keeping the resilience mechanics entirely invisible at the `ProviderClient` interface boundary — `PaymentService` and `RefundService` needed zero changes. `ProviderRegistry` wraps every resolved client transparently at construction time.
- 3 new test classes (`ChargeStoreTest`, `ChaosFilterTest` — using an injected, mockable `Random` bean rather than `ThreadLocalRandom.current()` inline, specifically so latency/failure rolls are deterministic in tests — and `ResilientProviderClientTest`, using real Resilience4j `Retry`/`CircuitBreaker` instances built with fast, test-tuned parameters rather than mocking Resilience4j's own types). The circuit-breaker test loops until the breaker actually reports `OPEN` rather than assuming a fixed call count trips it, and proves the delegate was skipped on the next call by asserting the exception's cause is `CallNotPermittedException` — a more robust signal than counting mock invocations, which turned out to be sensitive to exactly which call inside a batch crosses Resilience4j's internal threshold.
- Manually verified against the full live stack: confirmed idempotent charge creation (a repeated `merchantReference` returns the same `chargeId`); confirmed latency simulation (a measured ~1.8s round trip inside the configured range) and that `/actuator/health` stays completely unaffected by chaos even at a 100% failure rate; ran a full create→authorize→capture→refund flow with chaos at its deterministic defaults to confirm `ResilientProviderClient` doesn't break the normal path; at a 50% simulated failure rate, 8/8 authorize calls still succeeded, and Resilience4j's own Prometheus metrics confirmed why: `successful_with_retry: 5` — real recoveries, not luck; at a 100% failure rate, response times dropped from ~600-880ms (full 3-attempt retry cycles) to ~180-320ms (fast-fail) after the first few calls, and Prometheus confirmed the circuit breaker's state transitioning `closed` → `open` → `half_open` (an automatic recovery probe, per `automatic-transition-from-open-to-half-open-enabled`) as the simulated outage continued.

### Added (M13 — Deployment)
- [ADR-0013](docs/adr/0013-deployment-architecture.md): pins down the deployment target ADR-0012 only gestured at ("Docker Compose on Hetzner/DigitalOcean") — a single VPS, Nginx as the sole public-facing reverse proxy (only container exposing host ports), Let's Encrypt/certbot for TLS, and an honestly-scoped "CI/CD" — GitHub Actions publishes versioned images to GHCR on merge to `main`; there is no further deploy step because this project owns no live server to deploy to. `docs/DEPLOYMENT.md` documents the real remaining step (`pull` + `up -d` on an actual VPS) as a manual operation using the images CI publishes.
- `backend/payflow-core/Dockerfile` and `backend/mock-provider-service/Dockerfile`: multi-stage Maven build + `eclipse-temurin:21-jre-alpine` runtime, non-root user. Both need the *whole* `backend/` reactor as their build context (each module's parent is the reactor pom, so Maven can't resolve `-pl <module> -am` without every module's `pom.xml` present, even the one not being built).
- `infra/nginx/`: `Dockerfile` (multi-stage - builds `admin-dashboard` with Node, serves the output via `nginx:1.27-alpine`) and `nginx.conf` (HTTP→HTTPS redirect + ACME webroot passthrough, `/v1/` and `/actuator/health` proxied to `payflow-core`, SPA served with a `try_files ... /index.html` client-side-routing fallback).
- `infra/docker-compose.prod.yml`: only `nginx` publishes host ports (80/443) - Postgres, Redis, Kafka, Prometheus, Jaeger, the OTel Collector, Grafana, and both application services are reachable only on the internal Compose network, unlike `docker-compose.yml` (dev), which host-exposes everything for access from host-launched jars. Every service defines an `image:` alongside its `build:` block, so `docker compose build` builds from source (what this milestone's own manual verification used) while `docker compose pull` fetches the CI-published GHCR image instead (what a real VPS deploy does, per ADR-0013) - no source checkout needed on the deploy host for that path. Required secrets use Compose's `${VAR:?message}` interpolation, which refuses to even start the stack with a clear error if a secret is missing, rather than letting a container crash-loop.
- **Fixed a real gap in `mock-provider-service`:** it had only one `application.yml`, with a checked-in dev fallback default for `MOCK_PROVIDER_WEBHOOK_SECRET` (`dev-mock-webhook-secret-3f8a91c2`) that would have silently applied in production too - a publicly-known HMAC secret verifying inbound webhooks. Added `application-prod.yml` with no fallback, matching `payflow-core`'s own `application-prod.yml` discipline from M0. Verified both directions by hand: boots and 500s in `UnsatisfiedDependencyException`/`Could not resolve placeholder` without the env var, boots clean with it supplied. Caught the first time only because a *stale* locally-built jar (from before this fix) masked it entirely - re-running against a freshly `mvn package`d jar was what actually surfaced the failure.
- **Fixed a second real gap:** CI only ever built/tested the backend - the frontend (added M12) had no CI gate at all. `.github/workflows/ci.yml` gained a `frontend-build-lint` job (`npm ci && npm run lint && npm run build`), and `publish-images` now depends on both it and the existing backend job before publishing anything.
- **Fixed a third real gap, closing one M10 explicitly disclosed at the time:** M10's CHANGELOG entry for Grafana's anonymous-viewer access noted "a real internet-facing deployment would need network isolation or scrape-time credentials in front of it." `docker-compose.prod.yml` gives Grafana no public port at all; `nginx.conf`'s `/grafana/` location is the only path to it, gated by HTTP Basic Auth (`docs/DEPLOYMENT.md` documents generating `.htpasswd`) - the M12 dashboard's Metrics page keeps working (it embeds this same path), now behind a real access gate instead of the open-to-anyone dev default.
- **Found and fixed a real infinite-redirect-loop bug during this milestone's own manual verification, not by inspection:** `nginx.conf`'s original `/grafana/` block used `proxy_pass http://grafana:3000/;` (trailing slash), which strips the `/grafana` prefix before forwarding - but Grafana, configured with `GF_SERVER_SERVE_FROM_SUB_PATH: "true"` (needed for its own asset/link URLs to resolve correctly under the subpath), expects that prefix preserved on every request. The result was Grafana redirecting back to the `/grafana/...` URL it expected, Nginx stripping the prefix again on the retry, forever - `curl -L` confirmed it by hitting `--max-redirs`. Fixed by dropping the trailing path from `proxy_pass` (forwards the original URI unmodified); reverified with `curl -L` (0 redirects, 200) and confirmed the returned HTML's `<base href="/grafana/">` tag correctly anchors every relative asset URL, then fetched one of those asset URLs directly to prove it round-trips, not just that the page loads.
- Root-level and `backend/`-level `.dockerignore` (the latter already existed implicitly via context; the former didn't, and without it the `nginx` image's build context - repo root, since it needs both `frontend/admin-dashboard/` and `infra/nginx/nginx.conf` - was 108MB instead of the ~3KB it should be).
- Manually verified the entire production stack locally end to end (self-signed certificate standing in for a real one - see the honesty note in `docs/DEPLOYMENT.md`): all three images built clean; every service came up healthy with only `nginx` bound to a host port (confirmed via `docker compose ps`); a full signup → login → merchant → provider account → API key → create → authorize → capture → dashboard-summary flow succeeded end to end through Nginx's TLS termination and `/v1/` proxy, fully containerized on both ends (payflow-core and mock-provider-service talking to each other over the internal Compose network, not host-launched jars); Prometheus confirmed scraping both application services successfully via their container DNS names (`prometheus.prod.yml`, distinct from dev's `host.docker.internal`-based config); HTTP→HTTPS redirect confirmed; `/grafana/` confirmed 401 without credentials and reachable with them, including the redirect-loop fix above. Transient `NOT_COORDINATOR`/`UNKNOWN_TOPIC_OR_PARTITION` log lines during Kafka's own startup window and Grafana's SQLite "database is locked" retries during its own init were both observed and confirmed self-resolving, not new regressions - both are normal first-boot behavior for those images, unrelated to this milestone's changes.
