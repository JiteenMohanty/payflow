# ADR-0013: Deployment Architecture

**Status:** Accepted
**Date:** 2026-07-09

## Context

M0-M12 built the whole system and a heavy local dev stack (`infra/docker-compose.yml`: Postgres, Redis, Kafka, Prometheus, Grafana, Jaeger, an OTel Collector), but every service so far has run either as a Testcontainers-backed test or a manually-launched host jar (`java -jar ...`) - nothing has ever been containerized, and there is no production-shaped compose file, reverse proxy, or CD pipeline. ADR-0012 already named the intended target in passing ("the deployment story (Docker Compose on Hetzner/DigitalOcean)"), but never made it a real decision with consequences - this ADR does that.

This project has no owned domain, no provisioned VPS, and no cloud account - it is a from-scratch build exercise, not a company standing up real infrastructure. Any deployment decision here has to be honest about that: the parts that don't require owned infrastructure (Dockerfiles, a production compose file, a reverse proxy config, an image-publishing pipeline) can be built and verified for real; the parts that do (a live TLS certificate, a running server reachable at a domain) cannot, and pretending otherwise would mean undemonstrated, unverifiable configuration.

## Decision

- **Target shape: Docker Compose on a single VPS**, not Kubernetes, not a managed container platform. Matches ADR-0012's own stated intent and PayFlow's actual scale (a modular monolith plus one small simulator service, per ADR-0001) - a single host running `docker compose -f infra/docker-compose.prod.yml up -d` is sufficient, and anything heavier (an orchestrator, a service mesh) would be solving a scaling problem this system doesn't have.
- **Nginx as the single public-facing reverse proxy**, terminating TLS and doing two jobs: serving the built admin-dashboard SPA as static files (with a client-side-routing fallback to `index.html`, since the SPA owns its own routes under `/organizations/**`), and proxying `/v1/**` to `payflow-core`. It is the *only* container exposing ports to the host (80/443) - every other service (Postgres, Redis, Kafka, Prometheus, Grafana, Jaeger, the OTel Collector, both application services) stays on the internal Compose network only. An operator reaches Grafana or the app services directly via `docker exec` or an SSH tunnel, not a public port - there is no case for exposing an idempotency-key Redis instance or an unauthenticated-by-default Grafana straight to the internet.
- **TLS via Let's Encrypt / certbot**, the standard free-and-automatable choice for a self-hosted single-domain deployment - no paid CA, no self-signed cert accepted for real traffic. Because this project has no owned domain to actually issue a certificate against, `infra/docker-compose.prod.yml`'s `nginx` service and `docs/DEPLOYMENT.md` are both written for the real certbot webroot flow, but local verification of the compose stack necessarily substitutes a self-signed certificate - documented as exactly that, not presented as equivalent to a verified Let's Encrypt issuance.
- **"CI/CD" honestly scoped to CI + continuous *delivery* of images, not continuous *deployment* to a live host.** GitHub Actions, on every merge to `main`, builds and pushes versioned Docker images (`payflow-core`, `mock-provider-service`, and the Nginx+frontend image) to GitHub Container Registry (GHCR) - authenticated via the workflow's own `GITHUB_TOKEN`, no external registry account needed. It stops there: there is no SSH-deploy step, because there is no server to SSH into. `docs/DEPLOYMENT.md` documents the remaining, real-world step (`docker compose pull && docker compose up -d` on the actual VPS) as what an operator does with the images this pipeline publishes - a correct, literal continuation of the pipeline that this project simply cannot execute against infrastructure it doesn't own.
- **Secrets via a `.env` file on the deploy host, never committed.** `application-prod.yml` (payflow-core) already requires every secret via `${VAR}` with no fallback, from M0 - `docker-compose.prod.yml` and `docs/DEPLOYMENT.md` extend that same discipline to `mock-provider-service` (which had a real gap: its only `application.yml` shipped a checked-in dev webhook secret as a fallback default that would have silently applied in prod too - fixed alongside this ADR) and to the Postgres/Grafana admin credentials.

## Alternatives Considered

- **Kubernetes / a managed container platform (ECS, Cloud Run, etc.)**: rejected - solves a horizontal-scaling and multi-node-scheduling problem this system, at its current and realistically foreseeable scale (per ADR-0001's own modular-monolith reasoning), does not have. Would add real operational surface (a control plane, RBAC, ingress controllers) for no corresponding benefit.
- **A PaaS (Render, Railway, Fly.io)**: rejected for the same reason ADR-0012 gave for not using a hosted observability SaaS - it would make the deployment story dependent on an external platform's specific conventions rather than a portable, self-contained Compose file that runs identically on any VPS.
- **Self-signed certificates presented as the real answer, skipping Let's Encrypt entirely**: rejected - a self-signed cert is only acceptable as a stand-in for *local verification of this ADR's config*, not as production guidance; every real single-domain self-hosted deployment should get a real, browser-trusted certificate, which Let's Encrypt makes free and automatable.
- **A full SSH-based CD step that deploys to a placeholder/example host**: rejected - would produce a GitHub Actions step that looks like it deploys somewhere but has never actually run against real infrastructure, which is worse than clearly stopping at "publishes images" and documenting the manual step honestly.

## Consequences

- Every image built in CI must actually run correctly under `docker compose -f infra/docker-compose.prod.yml up` locally, since that is the only way this project can verify the deployment story end to end without owned infrastructure - "builds" is not the bar, "the whole stack boots and a real payment flow works through Nginx" is.
- Because `mock-provider-service` is deliberately not a real production dependency in any real deployment of this system (a genuine external provider would replace it - see EDD §2), including it in `docker-compose.prod.yml` at all is itself a scope decision: this project has no real provider integration to deploy instead, so `docker-compose.prod.yml` deploys the same architecture it always has, Mock Provider included, honestly labeled as such rather than pretending a real provider adapter exists.
- Adding a second provider adapter later (per EDD §14's "Future Extensibility") would mean `docker-compose.prod.yml` gains a real provider's credentials as additional deploy-host secrets, not a new deployment model - the Nginx/Compose/GHCR shape doesn't change.
- `docs/DEPLOYMENT.md` is the one document in this repository describing a process that cannot be fully verified by running it - it must say so plainly wherever that's true, rather than reading as if it had been.
